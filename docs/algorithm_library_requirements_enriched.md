# Algorithm Library Requirements — Version 4

## What changed in this version

This version keeps the full library from Version 3 and adds a new explicit implementation-spec column.

- In the main table the new column is called **Algorithm details**
- In the combination-method table the analogous new column is called **Method details**

These columns now state, in a much more explicit way:
- what the required inputs are
- which parameters should be exposed
- what the main algorithm steps are
- what outputs the implementation should produce
- what practical implementation notes the programmer should keep in mind

## Scope

- Algorithm rows: **121**
- Combination-method rows: **12**
- No rows were removed

## Recommended common interface

- Accept normalized market data in the same format as your existing algorithms
- Accept a structured parameter object / config
- Reuse shared indicator, statistics, and report helpers
- Return raw intermediate metrics or indicator series
- Return final normalized state per bar or time point: `buy`, `sell`, `neutral`
- Return optional confidence, score, or reason codes
- Return a report payload compatible with your current dashboard
- Make it possible to compose the strategy later using AND / OR / weighting / regime gating

## Implementation guidance for the coder

- Keep the existing registration and discovery mechanism so new algorithms appear automatically in the dashboard.
- Expose a machine-readable parameter schema for every algorithm and combination method.
- Keep outputs normalized so strategies can be composed later without rewriting the signal contract.
- Preserve raw intermediate series in reports whenever possible so the dashboard can explain why a signal changed.
- For strategies that are not naturally bar-by-bar (for example cross-sectional, factor, event-driven, or execution algorithms), keep the output format normalized anyway, but make the timing model explicit.
