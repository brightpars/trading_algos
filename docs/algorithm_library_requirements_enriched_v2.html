Home suitability scale: 1 = very suitable at home with relatively strong hardware; 2 = possible but materially harder because of data/execution complexity; 3 = generally a poor fit for home deployment because of latency, venue access, capital, or infrastructure needs.

Important validity notes:

  • There is no single canonical “complete list” of all trading algorithms; this catalog is intentionally broad and organized by family.
  • Advanced? and Home suitability are practical implementation judgments for your use case, not literal claims quoted from a source.
  • Initial reference is the starting source used to place a strategy into this catalog. For many items, the exact named variant has a broader literature beyond that starting point.

Recommended common interface

  • Accept normalized market data in the same format as your existing algorithms
  • Accept a structured parameter object / config
  • Reuse shared indicator/statistics/report helpers
  • Return raw intermediate metrics / indicator series
  • Return a final normalized state per bar or time point: buy, sell, neutral
  • Return optional confidence / score and reason codes
  • Return report payload compatible with the current dashboard

Child signals, timestamps, optional precedence rules Single composite discrete signal Implement this as a generic combiner over normalized child outputs. Align all child signals on the same timestamps, map `buy/sell/neutral` to a compact internal representation, and evaluate a configurable truth table such as AND, OR, k-of-n majority, or veto logic. The coder should define clear conflict handling, stale-signal handling, and precedence rules so the output is deterministic. This is the simplest reliable combination method and a good baseline for testing more advanced ensembles. CFA-ENS
2 Rule-Based Combination Weighted Linear Score Blend No All horizons; especially good for swing / daily systems 1 Normalize each child score and combine them with fixed or dynamic weights into one master score. Child scores, weights, normalization method, thresholds Composite numeric score then buy/sell/neutral Implement this by first normalizing every child model output to a common range, such as z-scores, bounded scores, or calibrated probabilities, and then computing a weighted sum. Convert the composite score to `buy/sell/neutral` with configurable thresholds and dead zones. Support both static weights and dynamic weights from a simple regime map or recent validation performance. The report should show each child contribution, the final blended score, and the exact threshold logic used to trigger a trade. CFA-ENS
3 Rule-Based Combination Rank Aggregation No Cross-sectional strategies / portfolio rotation 1 Convert child models into ranks and aggregate the ranks to reduce scale mismatch across models. Per-asset ranks from child strategies, aggregation rule Final rank or selection list Implement this when child strategies live on different scales or produce only relative preferences. Convert each child output into a cross-sectional rank, aggregate the ranks with average-rank, median-rank, or weighted-rank logic, and select the final portfolio or watchlist from the aggregate ordering. This method is robust when one model outputs probabilities and another outputs factor scores, because ranking removes scale mismatches. Expose per-child ranks and the final aggregate rank in the report. CFA-ENS
4 Optimization-Based Constrained Multi-Factor Optimization Yes Long-term and medium-term portfolio construction 2 Optimize combined exposure to several factors or signals while controlling active risk, beta, turnover, and concentration. Signal exposures, risk model, covariance matrix, optimization constraints Optimized portfolio weights Implement this as a bottom-up optimizer. Start from child signals or factor exposures, define the objective—such as maximizing combined exposure per unit of active risk—and then solve for weights under explicit constraints: turnover, active risk, beta, sector or country neutrality, concentration limits, liquidity, and maximum name weights. This is the most defensible way to combine several medium-term styles into one portfolio when you have a covariance model. The report should include the chosen weights, the objective value, risk statistics, turnover estimate, and which constraints were binding at the solution. MSCI-COREMF
5 Risk Overlay Risk Budgeting / Risk Parity No Portfolio-level combination across horizons 1 Allocate capital so no single strategy sleeve dominates total portfolio risk. Sleeve returns, volatility/covariance estimates, risk targets Portfolio weights or sleeve weights Implement this at the sleeve level rather than the single-security level. Estimate each child strategy’s volatility and covariance, then solve for weights so each sleeve contributes a balanced share of total portfolio risk. Support optional leverage caps, floor weights, and rebalance bands to limit turnover. The key reporting outputs are current sleeve weights, target risk contributions, realized risk contributions, and how far the live portfolio has drifted from its target risk budget. AQR-RP
6 Risk Overlay Volatility Targeting Overlay No All horizons; especially portfolio and sleeve management 1 Scale combined positions up or down to target a stable realized volatility level. Strategy returns, volatility estimate, target-vol setting Risk-scaled position multiplier Implement this as an overlay that rescales the gross exposure of a strategy or combination of strategies. Compute realized or forecast volatility on a rolling window, compare it with the desired target, and multiply positions by a scaling factor subject to caps, floors, and smoothing. This method is simple but operationally important because it stabilizes risk across time. The report should show forecast vol, target vol, applied multiplier, and any guardrails that clipped the scaling factor. AQR-STYLE
7 Adaptive / State-Based Regime Switching / HMM Gating No All horizons; especially when strategies behave differently in trend/range/high-vol regimes 2 Detect market regime first and activate only the strategies that historically work best in that regime. Returns, volatility, macro or market-state features, regime model Regime label plus gated strategy weights/signals Implement this as a two-stage controller. Stage one estimates regime—trend, range, crash, calm, high-volatility, risk-on, risk-off, or any custom taxonomy—using returns, volatility, breadth, or macro features. Stage two maps each detected regime to a set of active child strategies and their weights. Hidden Markov models, state-space models, or simpler classifiers can all work as the regime engine. The report should make the regime path and the strategy-activation map explicit so users can diagnose why a strategy was enabled or disabled. CFA-RL
8 Machine Learning Ensemble Bagging Ensemble No Daily to medium-term predictive systems 2 Train multiple versions of a model on resampled data and average their outputs to reduce variance and overfitting. Training data, bootstrap samples, child learner, validation setup Averaged forecast or vote Implement this on top of any probabilistic or score-based child model. Create bootstrap samples or resampled training folds, train one learner on each sample, and average the resulting predictions. Bagging is most useful when the base learner is noisy or unstable, because averaging reduces variance. The coder should keep out-of-sample evaluation separated cleanly from training and expose ensemble spread or model disagreement as an additional confidence metric. CFA-ENS
9 Machine Learning Ensemble Boosting Ensemble No Daily to medium-term predictive systems 2 Train models sequentially so each new learner focuses on errors made by prior learners. Training data, weak learner, loss function, tuning parameters Boosted forecast score Implement this as a sequential ensemble in which each learner focuses on the residual error or loss left by earlier learners. Gradient boosting trees are the most practical default if you are combining tabular financial features. Because boosting can overfit, the implementation should include early stopping, learning-rate controls, depth limits, and walk-forward validation. The output should include feature importance or SHAP-style diagnostics if available so the dashboard remains explainable. CFA-ENS
10 Machine Learning Ensemble Stacking / Meta-Learning No Daily to medium-term predictive systems 2 Feed several child model outputs into a second-stage model that learns when each model is trustworthy. Out-of-fold child predictions, meta-features, meta-model Meta-model forecast or final signal Implement this as a meta-model over child strategy outputs. First generate out-of-fold predictions from the child models so the meta-model sees realistic rather than in-sample inputs. Then train a second-stage model—linear, tree-based, or probabilistic—to learn when each child strategy is more trustworthy. This is usually the strongest general-purpose combination method when you have enough clean training data. The report should show child outputs, meta-features, meta-model prediction, and the contribution of each child model. CFA-ENS
11 Reinforcement Learning RL Allocation Controller Yes Portfolio allocation and execution over repeated decisions 3 Use reinforcement learning to allocate among strategies or adjust exposures based on state, action, and reward feedback. State features, reward function, action space, simulator or historical environment Dynamic weights / actions over time Implement this only if you are comfortable with simulation quality and governance. Define the state as current positions, recent returns, volatility, regime indicators, and possibly macro features; define the action as strategy weights or exposure changes; and define the reward as a risk-adjusted PnL function net of costs. Train in a controlled environment with realistic friction and position constraints, then evaluate with strict walk-forward testing. The report should capture the observed state, chosen action, reward, and any safety override that clipped the action. CFA-RL
12 Reinforcement Learning Hierarchical Controller / Meta-Policy Yes Complex multi-layer systems and multi-horizon control 3 Use a top-level controller to choose regime, gross exposure, or strategy family and a lower layer to pick concrete allocations or actions. State features, hierarchical action design, reward functions, child-policy outputs Layered allocation or action plan Implement this as a layered policy architecture. The top layer chooses a high-level mode—such as regime, gross-risk budget, or strategy family—while the lower layer translates that decision into concrete child-strategy weights or orders. This approach is valuable when you want clear separation between strategic and tactical decisions. Logging should be explicit: what the parent policy decided, what the child policies did, and how the final action was formed after constraints and risk guards were applied. CFA-RL

Implementation guidance for the coder

  1. Reuse-first: inspect the already-implemented algorithms and identify the exact base class/interface, indicator helpers, report builders, parameter validation pattern, and registration/discovery mechanism. New algorithms should fit that same architecture.
  2. Parameter schema: expose a machine-readable parameter schema for every algorithm so the dashboard can generate forms, validate input, save configs, and compose algorithms.
  3. Output normalization: even when internal math differs, algorithms should emit normalized outputs so they can be combined by AND / OR / pipeline / voting logic.
  4. Composition readiness: wherever reasonable, let the algorithm produce both a hard signal and an optional soft score so it can be reused inside composite strategies.
  5. Backtest transparency: always expose intermediate series used by the signal so the dashboard can explain why a trade was triggered.
  6. Point-in-time discipline for event/factor models: for earnings, fundamentals, and rebalance-event strategies, explicitly model the timestamp when data became knowable to avoid leakage.
  7. Universe-aware implementation: the new factor-style rows are mostly cross-sectional portfolio algorithms, so they should be implemented as universe-level selectors/weight builders rather than single-candle triggers.
